## Build & workflow commands

- Build everything (all nodes): `./gradlew chiseledBuild` ‚Äî this is the definition of "it builds." Registered in `stonecutter.gradle.kts`.
- Build one node: `./gradlew :1.21.1-neoforge:build`
- Run client for a node: `./gradlew :1.21.1-neoforge:runClient` (headless environments cannot verify this ‚Äî see Verification below)
- Switch active dev version: `./gradlew "Set active project to 1.21.1-fabric"` (or the IntelliJ Stonecutter plugin button). Do **not** call `stonecutterSwitchTo*` lifecycle tasks directly ‚Äî the docs mark them internal.
- **The first Gradle run downloads a large chunk of the modding ecosystem and can sit silent for many minutes. It is not hung. Do not kill it and do not "fix" it.**

## Stonecutter rules (violating these corrupts the working tree)

1. **Always return to the VCS version (`1.21.1-neoforge`) before committing.** Switching versions rewrites preprocessor comments across the codebase; committing from a non-VCS active version produces a garbage diff. Check `git status` after switching back ‚Äî it should be clean apart from your intended edits.
2. **Edit only the active branch of a preprocessor comment.** Inactive branches (`/*...*/` bodies) are regenerated by Stonecutter on switch. Never hand-align both branches; write the active one, switch, verify the other compiles.
3. Never edit anything under `build/` or `build/stonecutter-cache/`.
4. Comment syntax quick reference:
   - Line: `//? if neoforge` (next line only)
   - Block: `//? if >=1.21.4 {` ... `//?} else {` ... `//?}`
   - Inline: `method(/*? if fabric {*/ arg /*?}*/)`
   - Loader constants `fabric` / `neoforge` are registered via `constants.match` in the build config; version predicates compare against the node's Minecraft version.
5. Per-node dependency versions live in `versions/{node}/gradle.properties`. Global properties with `[VERSIONED]` markers in root `gradle.properties` are overridden there.

## Architecture rules (from PLAN.md ¬ß3 ‚Äî enforced in review)

1. **Architectury API first**: registries via `DeferredRegister`, events/networking/client registries via Architectury. If Architectury abstracts it, use it.
2. **Service layer for the gaps**: anything Architectury doesn't cover (data attachments especially) goes behind an interface in `platform/` with loader impls in small, individually-guarded files. Common code must never import `net.minecraft.fabric.*`, `net.fabricmc.*`, or `net.neoforged.*` outside `platform/`.
3. **Version shims**: version-conditional API differences route through shim classes, not scattered inline guards in gameplay code.
4. **Mixins are last resort**: prefer events/API hooks. Any mixin needs a comment stating exactly what breaks without it, and must be checked against both loaders.

## Verification & definition of done

An agent session may declare a task done only when:

1. `./gradlew chiseledBuild` passes locally.
2. Any GameTests touched or added pass on both loaders.
3. `git status` is clean on the VCS version and the diff contains only intended changes.

**What agents cannot verify:** anything visual or in-game (`runClient` behavior, rendering, GUIs, particles, creative-tab contents). Do not claim these work. Instead, end the session with a **human smoke-test list**: exact steps for Jacob to verify in-game, per loader. Compile-time success + a smoke-test list is the correct handoff; a claim of visual correctness is not.

## Git conventions

- Milestones are tagged `m0`, `m1`, ... only after the human smoke test passes.
- Small, single-purpose commits; message style: `system: what changed` (e.g., `aura: budgeted diffusion ticker`).
- Never commit with a non-VCS version active (rule 1 above).
